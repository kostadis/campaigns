This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## What This Repo Is

A multi-campaign D&D 5e workspace. Each subdirectory is a separate campaign managed with the **CampaignGenerator** toolkit (`~/CampaignGenerator/` or `~/src/CampaignGenerator/`). This repo contains no application code — it holds campaign documents, session data, and configuration.

`gm-assistant/` is **not** a campaign — it is workspace-scoped tooling and the canonical home of the `gm-*` Claude Code skills (`gm-session-prep`, `gm-faction-network`, `gm-strategy-doc`, `gm-handout`, `gm-npc-build`, `gm-npc-voice`, `gm-improv`, `gm-statblock`, `gm-social-encounter`). The entries under `.claude/skills/gm-*` are relative symlinks back to `gm-assistant/skills/`; edit the files in `gm-assistant/skills/`, not the symlinks. Changes here can ship in a single commit (they do not belong to any one campaign).

## Campaign Isolation (hard rule for commits and PRs)

**Each campaign evolves independently. Never bundle changes from multiple campaigns into a single commit or PR.** The campaigns are separate D&D games with separate states, schedules, players, and review cadence — mixing them couples unrelated work and pollutes per-campaign history.

Operationally:

- Before every commit, run `git status` and check directory prefixes. If files are touched across more than one of `Phandalin/`, `out-of-the-abyss/`, `stormgiants/`, `Hillsfar/`, `toee/`, **split into separate commits** — one per campaign.
- This holds even when the cross-campaign drift came "for free" from prior pipeline runs and the user said "commit everything."
- One campaign per PR. If accumulated drift spans campaigns, ask the user which campaign's changes ship first and commit only those.
- The only files that legitimately span campaigns are root-level shared infrastructure (`MEMPALACE_HOWTO.md`, this `CLAUDE.md`, `.gitignore` at repo root). Those can be their own small commit.

## Shared Architecture

Every campaign follows the same structure:

- `config.yaml` — CampaignGenerator workspace config (agent prompts, document paths, mempalace settings)
- `docs/` — grounding documents: `campaign_state.md`, `world_state.md`, `mechanics.md`, `planning.md`, `party.md`
- `summaries/` — raw session data organized by date (`YYYYMMDD/`), contains VTT transcripts
- `voice/` — per-character voice files (authoritative reference for how characters speak)
- `characters/` — character sheets and mechanical stats
- `examples/` — style example files for the session_doc pipeline
- `logs/` — operational logs from CampaignGenerator runs
- `.mcp.json` — MCP server registration (project-scoped)

### Campaigns with mempalace (Phandalin, Out of the Abyss, stormgiants)

Additional structure for semantic search:

- `mempalace.yaml` — wing/room architecture for the campaign reference wing
- `docs/chapters/` — bible split into per-chapter files (narrative wing)
- `docs/distill_extractions/` — LLM-generated structured extractions (chronicle wing)
- `entities.json` — generated by `mempalace init` (informational, not hand-edited)

See `MEMPALACE_HOWTO.md` at repo root for the full setup guide.

## CampaignGenerator Commands

All scripts live in `~/CampaignGenerator/` (or `~/src/CampaignGenerator/`). Run from inside a campaign directory so `config.yaml` is auto-detected.

```bash
# Session prep
python ~/CampaignGenerator/prep.py --beat "description of what happens"
python ~/CampaignGenerator/prep.py --mode pipeline --beat "..."   # three-agent pipeline
python ~/CampaignGenerator/prep.py --session "1. ... 2. ... 3. ..."  # full arc

# Regenerate grounding docs
python ~/CampaignGenerator/campaign_state.py summaries.md --output docs/campaign_state.md
python ~/CampaignGenerator/distill.py summaries.md --output docs/world_state.md
python ~/CampaignGenerator/party.py --character docs/characters/*.md --summaries summaries.md --output docs/party.md

# Convert D&D Beyond PDF character sheet
python ~/CampaignGenerator/dnd_sheet.py docs/sheet.pdf --output docs/characters/name.md
```

See `~/CampaignGenerator/CLAUDE.md` for the full script reference including `session_doc.py` (five-pass narrative pipeline), the Flask editor, and the Streamlit app.

## Mempalace Commands

```bash
MP=/home/kroussos/worldanvil_pipeline/venv/bin/mempalace

$MP init --yes <campaign_dir>          # generates entities.json (overwrites root mempalace.yaml!)
$MP mine <campaign_dir>/docs/chapters  # mine subdirs BEFORE root
$MP mine <campaign_dir>                # mine campaign reference wing
$MP search "query" --wing narrative    # search a specific wing
$MP status                             # shows all wings across all campaigns
```

**Order matters:** mine subdirectory wings before root to avoid double-mining. Root `.mempalaceignore` (or `.gitignore` fallback) excludes subdirs from root mining. Prefer `.mempalaceignore` to decouple mining exclusions from version-control concerns.

## MCP Servers

Each campaign registers MCP servers in `.mcp.json`:
- **campaign** — CampaignGenerator MCP server (session doc pipeline, prep)
- **mempalace** — semantic search over campaign content (where configured)
- **5etools** — D&D 5e reference data (monsters, spells, items, sections from published modules)

## Trust Hierarchy for Campaign Content

| Trust level | What | Rule |
|-------------|------|------|
| **Authoritative** | Session summaries, VTT transcripts, chapter splits | Verify here before building on a claimed fact |
| **Search accelerator** | `distill_extractions/`, `planning_extractions/` | Use to find the right time window, then verify against authoritative source |
| **Working reference** | Grounding docs (`world_state.md`, `planning.md`, NPC dossiers) | Use for building forward; verify against authoritative source when precision matters |

## Key Conventions

- **Voice files are authoritative** for character speech and personality. Always read the voice file before writing dialogue or narrative for a character.
- **Players are the ultimate authority** on their characters' psychology. When a player corrects a characterization, update the voice file.
- **Notes are staging, not canon.** Nothing enters the mempalace directly from `notes/`. Human review required before promotion to grounding docs.
- **Published modules stay outside the palace** when 5etools MCP is available — use `get_section` and `search` instead.
- Session summaries use chapter numbering (e.g., "Chapter 34: Where Cows Come Home").
- Session transcripts are VTT files in `summaries/YYYYMMDD/`.
